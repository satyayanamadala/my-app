01 — Framework

The DevSecOps Infinity Model

Traditional DevOps treats security as a final gate before release. DevSecOps integrates security as a shared responsibility across every phase — making security a feature of the development process, not a separate audit.

In healthcare, this matters critically: a single PHI exposure can result in multi-million dollar HIPAA penalties, loss of patient trust, and regulatory shutdowns.

02 — Threat Modeling

Healthcare Threat Landscape

Healthcare organizations are the #1 target for ransomware and data breaches globally. STRIDE threat modeling identifies attack vectors specific to clinical and patient data systems.

PHI Data Exfiltration
CRITICAL

Unauthorized access to patient records, insurance data, or diagnosis history via SQL injection, broken access control, or insider threats.

→ RBAC + column-level encryption + DLP
Ransomware Attack
CRITICAL

Malicious actors encrypt clinical systems, halting patient care. Healthcare systems are targeted 3× more than any other sector.

→ Network segmentation + immutable backups
Insecure API Endpoints
HIGH

HL7 FHIR APIs with missing authentication, broken object-level authorization, or verbose error messages exposing PHI in responses.

→ OAuth 2.0 + FHIR SMART scope validation
Secrets in Source Code
HIGH

Database credentials, API keys, or encryption keys committed to version control. Frequently exploited via public repository exposure.

→ HashiCorp Vault + GitLeaks pre-commit
Vulnerable Dependencies
MEDIUM

Third-party libraries with known CVEs included in production builds. Common in legacy healthcare systems with infrequent updates.

→ OWASP Dependency-Check + Trivy SCA
Misconfigured Cloud Resources
LOW

S3 buckets with public access, overly permissive IAM roles, or unencrypted RDS instances exposing PHI inadvertently.

→ AWS Config Rules + Checkov IaC scan
03 — Code Security

SAST & DAST Scanning

Static Application Security Testing (SAST) analyzes source code for vulnerabilities before runtime. Dynamic Application Security Testing (DAST) attacks a running instance to find exploitable issues. Both are required for HIPAA Technical Safeguards compliance.

Pre-commit Security Hooks

.pre-commit-config.yaml 
repos:
  # Detect secrets before they enter version control
  - repo: https://github.com/gitleaks/gitleaks
    rev: v8.18.0
    hooks:
      - id: gitleaks

  # Python SAST (bandit)
  - repo: https://github.com/PyCQA/bandit
    rev: 1.7.7
    hooks:
      - id: bandit
        args: ["-r", "src/", "-ll"]

  # Detect PHI patterns (custom regex)
  - repo: local
    hooks:
      - id: phi-detector
        name: PHI Pattern Detector
        entry: python scripts/phi_scan.py
        language: python
        types: [python, javascript, typescript]

OWASP ZAP DAST in Jenkins

Jenkinsfile — DAST Stage 
stage('DAST — OWASP ZAP') {
  steps {
    script {
      // Spin up ZAP proxy container
      sh """
        docker run --rm --network=test-net \\
          -v \$(pwd)/zap-reports:/zap/wrk/ \\
          ghcr.io/zaproxy/zaproxy:stable zap-api-scan.py \\
          -t http://app-staging:3000/openapi.json \\
          -f openapi \\
          -r zap-report.html \\
          -w zap-report.md \\
          -x zap-report.xml \\
          -I \\                        # Don't fail on warnings
          -z "-config scanner.maxScanDurationInMins=10"
      """

      // Parse XML for HIGH findings — fail build if found
      def zapReport = readFile('zap-reports/zap-report.xml')
      if (zapReport.contains('riskdesc="High')) {
        error('DAST: HIGH severity vulnerabilities found. Build blocked.')
      }

      // Archive reports
      publishHTML(target: [
        reportDir: 'zap-reports',
        reportFiles: 'zap-report.html',
        reportName: 'OWASP ZAP DAST Report'
      ])
    }
  }
}
04 — Secrets Management

HashiCorp Vault for PHI Systems

No credentials should ever exist in environment variables, Kubernetes Secrets (base64 only!), or configuration files. HashiCorp Vault provides dynamic secrets, automatic rotation, and a complete audit trail for every secret access — essential for HIPAA audit requirements.

⛔
HIPAA Violation Risk

Kubernetes Secrets are only base64-encoded, not encrypted by default. Always enable etcd encryption at rest and use Vault for any secret that touches PHI systems. Unencrypted secrets violate HIPAA Technical Safeguards §164.312(a)(2)(iv).

Vault Policy for Healthcare App

vault/policies/healthcare-app.hcl 
# Policy for healthcare application service account
# Principle of least privilege — only paths required

path "database/creds/ehr-readonly" {
  capabilities = ["read"]
  # Dynamic credentials — rotate every 1h
}

path "secret/data/healthcare/+/api-keys" {
  capabilities = ["read"]
}

path "pki/issue/healthcare-internal" {
  capabilities = ["create", "update"]
  # TLS cert issuance for mTLS between services
}

# Explicitly deny access to PHI encryption keys
path "transit/keys/phi-encryption" {
  capabilities = ["deny"]
}

# Allow encryption/decryption operations, NOT key export
path "transit/encrypt/phi-encryption" {
  capabilities = ["update"]
}

path "transit/decrypt/phi-encryption" {
  capabilities = ["update"]
}

Kubernetes Vault Agent Injector

k8s/deployment-annotated.yaml (excerpt) 
metadata:
  annotations:
    # Vault Agent sidecar injection
    vault.hashicorp.com/agent-inject: "true"
    vault.hashicorp.com/role: "healthcare-app"

    # Inject DB credentials as file (not env var)
    vault.hashicorp.com/agent-inject-secret-db: "database/creds/ehr-readonly"
    vault.hashicorp.com/agent-inject-template-db: |
      {{- with secret "database/creds/ehr-readonly" -}}
      DB_USER={{ .Data.username }}
      DB_PASS={{ .Data.password }}
      DB_HOST=ehr-db.internal:5432
      {{- end }}

    # TLS certificate for mTLS
    vault.hashicorp.com/agent-inject-secret-tls: "pki/issue/healthcare-internal"

    # Auto-renew credentials before expiry
    vault.hashicorp.com/agent-pre-populate-only: "false"
05 — Network Security

Network Segmentation & mTLS

All PHI-handling services must communicate exclusively over mutually-authenticated TLS (mTLS). Kubernetes NetworkPolicies enforce strict egress/ingress rules, preventing lateral movement in case of compromise.

NetworkPolicy — PHI Service Isolation

k8s/network-policy-ehr.yaml 
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: ehr-phi-isolation
  namespace: healthcare
spec:
  podSelector:
    matchLabels:
      data-class: phi
  policyTypes:
  - Ingress
  - Egress

  ingress:
  - from:
    # Only allow traffic from API gateway with mTLS cert
    - podSelector:
        matchLabels:
          role: api-gateway
          istio-mtls: "enabled"
    ports:
    - protocol: TCP
      port: 8443    # HTTPS only

  egress:
  - to:
    # Only allow connection to encrypted EHR database
    - podSelector:
        matchLabels:
          role: ehr-database
    ports:
    - protocol: TCP
      port: 5432    # Postgres TLS

  # Block ALL other ingress/egress (default deny)
06 — Compliance

HIPAA Technical Safeguards Mapping

Every HIPAA Technical Safeguard requirement must map to a concrete DevSecOps control implemented in the pipeline. The following table maps HIPAA §164.312 to specific tooling.

HIPAA Safeguard Section Implementation Status
Access Control — Unique user ID §164.312(a)(2)(i) Keycloak SSO + RBAC in K8s, no shared accounts ✓ PASS
Encryption & Decryption §164.312(a)(2)(iv) AES-256 at rest (etcd), TLS 1.3 in transit, Vault Transit ✓ PASS
Audit Controls §164.312(b) Splunk SIEM + Loki structured logging, 6-year retention ✓ PASS
Integrity — PHI alteration detection §164.312(c)(1) SHA-256 checksums on PHI records, immutable audit log ✓ PASS
Person or Entity Authentication §164.312(d) MFA enforced for all PHI access, FIDO2 hardware keys for admins ✓ PASS
Transmission Security §164.312(e)(2)(ii) mTLS Istio service mesh, TLS 1.3 minimum, no TLS 1.0/1.1 ✓ PASS
Emergency Access Procedure §164.312(a)(2)(ii) Break-glass procedures with enhanced audit logging + PagerDuty alert ⚠ REVIEW
Automatic Logoff §164.312(a)(2)(iii) 15-min session timeout for clinical UIs, 8h for background services ✓ PASS
07 — Identity & Access

IAM & RBAC Design

Role-Based Access Control must enforce the minimum necessary standard — each user role can access only the PHI required to perform their clinical function. HIPAA minimum necessary standard §164.514(d).

Healthcare Role Definitions

k8s/rbac/healthcare-roles.yaml 
# Clinician Role — read-only PHI, no admin access
apiVersion: rbac.authorization.k8s.io/v1
kind: ClusterRole
metadata:
  name: clinician
  labels:
    hipaa-role: "true"
    phi-access: "read-only"
rules:
- apiGroups: ["healthcare.io"]
  resources: ["patientrecords", "prescriptions"]
  verbs: ["get", "list"]
  # No create/update/delete without audit approval
---
# Pharmacist — specific prescription access
apiVersion: rbac.authorization.k8s.io/v1
kind: ClusterRole
metadata:
  name: pharmacist
rules:
- apiGroups: ["healthcare.io"]
  resources: ["prescriptions"]
  verbs: ["get", "list", "update"]
- # Denied: patientrecords, billing, admin
---
# DevOps Engineer — NO PHI access, infra only
apiVersion: rbac.authorization.k8s.io/v1
kind: ClusterRole
metadata:
  name: devops-engineer
rules:
- apiGroups: [""]
  resources: ["pods", "services", "deployments"]
  verbs: ["get", "list", "create", "update", "delete"]
- # Explicitly denied: patientrecords, PHI namespaces
👤

Zero Trust Architecture

Never trust, always verify. Every service-to-service call is authenticated via mTLS certificates issued by Vault PKI.

🔐

Just-In-Time Access

Elevated PHI access is granted on-demand with time limits (max 1h), full audit trail, and automatic revocation.

🔑

MFA Enforcement

All clinical staff require hardware FIDO2 keys or TOTP for any PHI access. Password-only auth is blocked at the IdP level.

📋

Quarterly Access Reviews

Automated reports flag unused roles and overprivileged accounts for quarterly review by Compliance and Security teams.

HIPAA mandates recording all access to PHI with who, what, when, and from where. Logs must be immutable, tamper-evident, retained for 6 years, and available within 24 hours of a request.

09 — CI/CD Security

Secure Jenkins Pipeline

The healthcare DevSecOps pipeline gates every deployment through mandatory security checks. No PHI-adjacent system can be deployed unless all security stages pass.

Complete Secure Pipeline

Jenkinsfile — Healthcare DevSecOps 
pipeline {
  agent { label 'secure-agent' }  // Hardened Jenkins agent

  environment {
    VAULT_ADDR   = credentials('vault-address')
    REGISTRY     = credentials('registry-url')
    SONAR_TOKEN  = credentials('sonarqube-token')
  }

  stages {

    stage('Secrets Detection') {
      steps {
        sh 'gitleaks detect --source . --exit-code 1'
        sh 'trufflehog git file://. --only-verified'
      }
    }

    stage('SAST — SonarQube') {
      steps {
        sh """
          sonar-scanner \\
            -Dsonar.projectKey=healthcare-ehr \\
            -Dsonar.sources=src/ \\
            -Dsonar.tests=tests/ \\
            -Dsonar.qualitygate.wait=true \\
            -Dsonar.security.hotspots=true
        """
      }
    }

    stage('PHI Pattern Scan') {
      steps {
        sh """
          # Scan for hardcoded SSN, MRN, DOB patterns
          python scripts/phi_scanner.py --path src/ \\
            --patterns config/phi-patterns.yaml \\
            --fail-on-match
        """
      }
    }

    stage('SCA — Dependency Check') {
      steps {
        sh """
          dependency-check --scan . \\
            --failOnCVSS 7 \\
            --format HTML,JSON \\
            --out reports/dependency-check/
        """
      }
    }

    stage('Build & Sign Image') {
      steps {
        script {
          sh "docker build -t ${REGISTRY}/ehr:${GIT_COMMIT[0..7]} ."
          // Sign with Cosign for supply chain integrity
          sh "cosign sign --key vault://transit/sign-key ${REGISTRY}/ehr:${GIT_COMMIT[0..7]}"
        }
      }
    }

    stage('Container Scan — Trivy') {
      steps {
        sh """
          trivy image \\
            --exit-code 1 \\
            --severity HIGH,CRITICAL \\
            --ignore-unfixed \\
            --format sarif \\
            --output reports/trivy.sarif \\
            ${REGISTRY}/ehr:${GIT_COMMIT[0..7]}
        """
      }
    }

    stage('OPA Policy Check') {
      steps {
        sh """
          # Validate K8s manifests against HIPAA OPA policies
          conftest test k8s/ \\
            --policy policies/hipaa/ \\
            --all-namespaces
        """
      }
    }

    stage('Deploy to Staging') {
      steps {
        sh 'helm upgrade --install ehr-staging ./helm/ehr --values values/staging.yaml'
      }
    }

    stage('DAST — OWASP ZAP') {
      steps {
        sh """
          docker run --rm --network=test-net \\
            ghcr.io/zaproxy/zaproxy:stable zap-api-scan.py \\
            -t http://ehr-staging:8080/openapi.json \\
            -f openapi -I -r reports/zap.html
        """
      }
    }

    stage('Compliance Gate') {
      steps {
        script {
          def report = readJSON file: 'reports/compliance-summary.json'
          if (report.criticalFindings > 0) {
            error("HIPAA Compliance FAILED: ${report.criticalFindings} critical issues")
          }
          echo "✅ Compliance Gate PASSED"
        }
      }
    }

    stage('Deploy to Production') {
      when { branch 'main' }
      input {
        message "Deploy to Production? Compliance officer approval required."
        submitter "compliance-officer,ciso"
      }
      steps {
        sh 'helm upgrade --install ehr ./helm/ehr --values values/production.yaml'
      }
    }
  }
}
10 — Incident Response

Security Incident Response

HIPAA Breach Notification Rule §164.400 requires notifying HHS and affected individuals within 60 days of discovering a breach. An automated incident response pipeline dramatically compresses detection-to-containment time.

1
Detection <5 min — SIEM Alert

Splunk/ELK detects anomalous PHI access patterns, unusual data export volumes, or failed auth spikes. PagerDuty notifies on-call security engineer within 30 seconds.

2
Containment <15 min — Automated Isolation

Runbook automation applies Kubernetes NetworkPolicy to isolate the compromised pod. Vault revokes dynamic credentials for the affected service account immediately.

3
Assessment <4 hrs — PHI Impact Analysis

Security team analyzes immutable audit logs to determine which patient records were accessed, by whom, and whether PHI was exfiltrated. HIPAA breach determination is made.

4
Notification <60 days — HHS + Patients

If PHI breach confirmed, HHS breach portal submission and affected patient notifications are dispatched. Internal root cause analysis documents remediation steps.

5
Recovery — Remediation + Hardening

Vulnerability is patched, security controls updated, pipeline policies tightened. Post-incident review updates threat model and OPA policies to prevent recurrence.
