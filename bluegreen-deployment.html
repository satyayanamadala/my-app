01 — Overview

What is Blue-Green Deployment?

Blue-Green deployment is a release technique that reduces downtime and risk by running two identical production environments. At any time, only one environment (say, Blue) serves live traffic while the other (Green) is idle, available for the new release.

Once deployment and smoke testing on Green succeeds, the load balancer instantly flips all traffic to Green. Blue then becomes the new staging environment. Rollback is a single DNS/LB switch.

ℹ
Key Principle

Both environments are identical in infrastructure. The ONLY difference is the application version running. This parity is enforced through Docker images and Kubernetes manifests stored in version control.

02 — Benefits

Why Blue-Green?

⚡

Zero Downtime

Traffic switchover happens at the load balancer level in milliseconds. Users experience no interruption during deployments.

↩

Instant Rollback

If issues arise post-deployment, flipping traffic back to the Blue environment takes under 5 seconds — no re-deployment needed.

🧪

Full Production Testing

Green environment runs on real production infrastructure before receiving live traffic. Test with production data via read-only access.

🔒

Reduced Risk

New releases are validated in an identical environment. The old version remains untouched until confidence is established.

🔄

Consistent Releases

The same automated pipeline is used for every release, eliminating manual steps and human error in the deployment process.

📊

Clear Audit Trail

Jenkins pipeline logs, Docker image tags, and Kubernetes deployment history provide full traceability for compliance.

03 — Infrastructure

Docker Configuration

Each application is containerized using Docker. Images are tagged with Git commit SHAs and semantic version labels, ensuring immutable, reproducible builds. Multi-stage builds reduce image size and eliminate build-time dependencies from runtime.

Production Dockerfile

Dockerfile 
# ── Stage 1: Build ──────────────────────────────────────
FROM node:18-alpine AS builder
WORKDIR /app

# Install dependencies first (layer cache optimization)
COPY package*.json ./
RUN npm ci --only=production

COPY . .
RUN npm run build

# ── Stage 2: Runtime ─────────────────────────────────────
FROM node:18-alpine AS runtime
WORKDIR /app

# Security: run as non-root user
RUN addgroup -S appgroup && adduser -S appuser -G appgroup
USER appuser

COPY --from=builder /app/dist ./dist
COPY --from=builder /app/node_modules ./node_modules

# Labels for traceability
LABEL maintainer="devops@company.com"
LABEL version="${BUILD_VERSION}"
LABEL git-commit="${GIT_COMMIT}"

EXPOSE 3000
HEALTHCHECK --interval=30s --timeout=10s --retries=3 \
  CMD wget -qO- http://localhost:3000/health || exit 1

CMD ["node", "dist/server.js"]

04 — Orchestration

Kubernetes Manifests

Kubernetes manages both the Blue and Green deployments as separate Deployment resources. A shared Service object uses label selectors to point to the active environment.

Blue Deployment

k8s/deployment-blue.yaml 
apiVersion: apps/v1
kind: Deployment
metadata:
  name: myapp-blue
  namespace: production
  labels:
    app: myapp
    slot: blue
    version: "1.4.2"
spec:
  replicas: 3
  selector:
    matchLabels:
      app: myapp
      slot: blue
  template:
    metadata:
      labels:
        app: myapp
        slot: blue
        version: "1.4.2"
    spec:
      containers:
      - name: myapp
        image: registry.io/myapp:1.4.2
        ports:
        - containerPort: 3000
        readinessProbe:
          httpGet:
            path: /health
            port: 3000
          initialDelaySeconds: 10
          periodSeconds: 5
        livenessProbe:
          httpGet:
            path: /health
            port: 3000
          initialDelaySeconds: 20
          periodSeconds: 10
        resources:
          requests:
            cpu: "100m"
            memory: "256Mi"
          limits:
            cpu: "500m"
            memory: "512Mi"

Shared Service — Traffic Switch via Label Selector

k8s/service.yaml 
apiVersion: v1
kind: Service
metadata:
  name: myapp-service
  namespace: production
spec:
  selector:
    app: myapp
    slot: blue   # ← Change to "green" to switch traffic
  ports:
  - protocol: TCP
    port: 80
    targetPort: 3000
  type: LoadBalancer
✓
Traffic Switch Command

kubectl patch service myapp-service -n production -p '{"spec":{"selector":{"slot":"green"}}}' — This single command atomically switches all traffic in under 2 seconds.

Jenkinsfile

Jenkinsfile 
pipeline {
  agent any
  environment {
    REGISTRY     = 'registry.company.io'
    IMAGE_NAME   = 'myapp'
    K8S_NS       = 'production'
    IMAGE_TAG    = "${BUILD_NUMBER}-${GIT_COMMIT[0..7]}"
  }

  stages {

    stage('Checkout') {
      steps {
        checkout scm
        script {
          env.GIT_COMMIT_MSG = sh(script: 'git log -1 --pretty=%B', returnStdout: true).trim()
        }
      }
    }

    stage('Build & Push Image') {
      steps {
        script {
          docker.withRegistry("https://${REGISTRY}", 'registry-creds') {
            def img = docker.build(
              "${REGISTRY}/${IMAGE_NAME}:${IMAGE_TAG}",
              "--build-arg BUILD_VERSION=${IMAGE_TAG} ."
            )
            img.push()
            img.push('latest')
          }
        }
      }
    }

    stage('Security Scan') {
      steps {
        sh """
          trivy image --exit-code 1 --severity HIGH,CRITICAL \\
            ${REGISTRY}/${IMAGE_NAME}:${IMAGE_TAG}
        """
      }
    }

    stage('Detect Active Slot') {
      steps {
        script {
          env.ACTIVE_SLOT = sh(
            script: """kubectl get svc myapp-service -n ${K8S_NS} \\
              -o jsonpath='{.spec.selector.slot}'""",
            returnStdout: true
          ).trim()
          env.DEPLOY_SLOT = (env.ACTIVE_SLOT == 'blue') ? 'green' : 'blue'
          echo "Active: ${ACTIVE_SLOT} → Deploying to: ${DEPLOY_SLOT}"
        }
      }
    }

    stage('Deploy to Inactive Slot') {
      steps {
        script {
          sh """
            kubectl set image deployment/myapp-${DEPLOY_SLOT} \\
              myapp=${REGISTRY}/${IMAGE_NAME}:${IMAGE_TAG} \\
              -n ${K8S_NS}
            kubectl rollout status deployment/myapp-${DEPLOY_SLOT} \\
              -n ${K8S_NS} --timeout=300s
          """
        }
      }
    }

    stage('Smoke Tests') {
      steps {
        script {
          def greenUrl = sh(
            script: """kubectl get svc myapp-${DEPLOY_SLOT}-internal \\
              -n ${K8S_NS} -o jsonpath='{.status.loadBalancer.ingress[0].ip}'""",
            returnStdout: true
          ).trim()
          sh """
            curl -sf http://${greenUrl}/health | grep -q '"status":"ok"'
            curl -sf http://${greenUrl}/api/ping | grep -q 'pong'
          """
        }
      }
    }

    stage('Switch Traffic') {
      steps {
        script {
          sh """
            kubectl patch service myapp-service -n ${K8S_NS} \\
              -p '{"spec":{"selector":{"slot":"${DEPLOY_SLOT}"}}}'
          """
          echo "✅ Traffic switched to ${DEPLOY_SLOT} (${IMAGE_TAG})"
        }
      }
    }
  }

  post {
    failure {
      script {
        // Auto-rollback on failure
        sh """kubectl patch service myapp-service -n ${K8S_NS} \\
          -p '{"spec":{"selector":{"slot":"${ACTIVE_SLOT}"}}}'"""
        slackSend channel: '#deployments',
          color: 'danger',
          message: "❌ Deployment FAILED for ${IMAGE_TAG}. Rolled back to ${ACTIVE_SLOT}."
      }
    }
    success {
      slackSend channel: '#deployments',
        color: 'good',
        message: "✅ Deployed ${IMAGE_TAG} to ${DEPLOY_SLOT} successfully."
    }
  }
}
07 — Recovery

Rollback Strategy

Because the previous environment remains intact and scaled, rollback is a pure traffic switch — not a re-deployment. This makes mean time to recovery (MTTR) dramatically lower compared to traditional rolling deployments.

Automatic Rollback Triggers

Error Rate > 1%
→
Alert Fires
→
Jenkins Job Triggered
→
K8s Service Patched
→
✓ Rolled Back

Manual Rollback Script

scripts/rollback.sh 
#!/bin/bash
set -euo pipefail

NS="production"
SVC="myapp-service"

# Detect current active slot
CURRENT=$(kubectl get svc $SVC -n $NS -o jsonpath='{.spec.selector.slot}')
TARGET=$( [[ "$CURRENT" == "blue" ]] && echo "green" || echo "blue" )

echo "Rolling back: ${CURRENT} → ${TARGET}"

# Switch traffic
kubectl patch service $SVC -n $NS \
  -p "{\"spec\":{\"selector\":{\"slot\":\"${TARGET}\"}}}"

# Verify
VERIFY=$(kubectl get svc $SVC -n $NS -o jsonpath='{.spec.selector.slot}')
echo "✅ Traffic now pointing to: ${VERIFY}"
08 — Observability

Monitoring & Health Checks

Observability is critical for safe deployments. The following stack provides full visibility into deployment health.

Tool Role Key Metrics Alert Threshold
Prometheus Metrics scraping Request rate, latency, error rate Error rate > 1% → rollback
Grafana Dashboards P50/P99 latency, pod restarts P99 > 500ms → alert
Alertmanager Alert routing Threshold breaches PagerDuty + Slack
Loki Log aggregation Error logs per slot Error spike → webhook
Jaeger Distributed tracing Trace spans per deployment slot Slow traces > 2s
⚠
Post-Switch Observation Window

After switching traffic to Green, maintain a 15–30 minute observation window before scaling down Blue. Monitor error rates, latency P99, and pod crash loops before considering the deployment complete.

09 — Guidelines

Best Practices

1
Database Migrations Must Be Backwards-Compatible

New schema changes must support both old and new code simultaneously. Use the expand-contract pattern: add columns before removing old ones.

2
Tag Images with Git SHA, Never Just "latest"

Use immutable tags like registry/app:1.5.0-a3f8c21 for full traceability and reproducibility of any deployed version.

3
Keep Both Slots Warm

Scale down idle slot replicas to 1 (not 0) to keep pods warm and ready for instant failover without cold start delays.

4
Smoke Tests Are Non-Negotiable

Automate critical user journey tests (login, key API endpoints, data retrieval) that must pass before any traffic switch is allowed.

5
Feature Flags Over Multiple Deployments

Combine blue-green with feature flags to decouple deployment from feature release — deploy often, release when business-ready.

6
Cost Management with HPA

Use Kubernetes HorizontalPodAutoscaler on the idle slot to scale down to minimum replicas during off-hours, reducing infrastructure costs.
